# Менеджер задач на Spring Boot

## Task 1: Разработка базовой функциональности

### Как запустить приложение

```bash
./mvnw spring-boot:run
```

После запуска приложения вы можете тестировать его API с помощью `curl`. Ниже приведены основные примеры команд для тестирования.

---

### Примеры команд для тестирования API

#### 1. Регистрация пользователя

```bash
curl -X POST -H "Content-Type: application/json" -d "{\"username\":\"user1\",\"password\":\"password123\"}" http://localhost:8080/api/users/register
```

#### 2. Авторизация (логин) пользователя

```bash
curl -X GET "http://localhost:8080/api/users/login?username=user1&password=password123"
```

#### 3. Создание задачи

```bash
curl -X POST -H "Content-Type: application/json" -d "{\"title\":\"Тестовая задача\",\"description\":\"Описание тестовой задачи\",\"targetDate\":\"2024-12-31T12:00:00\",\"userId\":\"ПОЛУЧЕННЫЙ_UUID_ПОЛЬЗОВАТЕЛЯ\"}" http://localhost:8080/api/tasks
```

#### 4. Получение всех задач пользователя

```bash
curl -X GET "http://localhost:8080/api/tasks?userId=ПОЛУЧЕННЫЙ_UUID_ПОЛЬЗОВАТЕЛЯ"
```

#### 5. Получение только невыполненных задач пользователя

```bash
curl -X GET "http://localhost:8080/api/tasks/pending?userId=ПОЛУЧЕННЫЙ_UUID_ПОЛЬЗОВАТЕЛЯ"
```

#### 6. Удаление задачи (отметка как удалённая)

```bash
curl -X DELETE "http://localhost:8080/api/tasks/ПОЛУЧЕННЫЙ_UUID_ЗАДАЧИ"
```

#### 7. Получение всех уведомлений пользователя

```bash
curl -X GET "http://localhost:8080/api/notifications?userId=ПОЛУЧЕННЫЙ_UUID_ПОЛЬЗОВАТЕЛЯ"
```

#### 8. Получение непрочитанных уведомлений пользователя

```bash
curl -X GET "http://localhost:8080/api/notifications/pending?userId=ПОЛУЧЕННЫЙ_UUID_ПОЛЬЗОВАТЕЛЯ"
```

#### 9. Отметка уведомления как прочитанного

```bash
curl -X POST "http://localhost:8080/api/notifications/ПОЛУЧЕННЫЙ_UUID_УВЕДОМЛЕНИЯ/read"
```

---

> **Примечание:** Во всех командах замените:
>
> * `ПОЛУЧЕННЫЙ_UUID_ПОЛЬЗОВАТЕЛЯ` — на реальный UUID пользователя, полученный после регистрации;
> * `ПОЛУЧЕННЫЙ_UUID_ЗАДАЧИ` — на UUID задачи, полученный при создании задачи;
> * `ПОЛУЧЕННЫЙ_UUID_УВЕДОМЛЕНИЯ` — на UUID уведомления, полученный в соответствующем ответе API.

## Task 2: Юнит-тестирование

### Запуск тестов

Для запуска тестов и генерации отчета о покрытии используйте команду:

```bash
./mvnw clean test jacoco:report
```

### Отчеты о покрытии кода тестами

После выполнения команды будет сгенерирован отчет о покрытии кода тестами, доступный по пути:

```
target/site/jacoco/index.html
```

Вы можете открыть этот файл в любом веб-браузере для просмотра подробной информации о покрытии кода:

- Общее покрытие по инструкциям, ветвлениям и методам
- Детальная информация по каждому пакету и классу
- Визуальное отображение покрытого и непокрытого кода

### Настройка порога покрытия

В проекте настроена проверка минимального покрытия кода на уровне 80%. Эта проверка выполняется при запуске тестов с помощью JaCoCo. Если покрытие падает ниже этого порога, сборка не будет успешной.

Настройки порога покрытия находятся в `pom.xml` в разделе конфигурации плагина JaCoCo:

```xml
<rule>
    <element>BUNDLE</element>
    <limits>
        <limit>
            <counter>INSTRUCTION</counter>
            <value>COVEREDRATIO</value>
            <minimum>0.80</minimum>
        </limit>
    </limits>
</rule>
```

## Task 3: Профили приложения

В проекте настроены 2 профиля для разных сред выполнения:

### Доступные профили

1. **dev** - профиль разработки с in-memory хранилищем данных
   - Хранит данные в памяти

2. **prod** - продакшн профиль с постоянным хранилищем данных на базе JPA
   - Использует H2 базу данных для хранения информации

### Запуск с определенным профилем

Для запуска приложения с выбранным профилем используйте:

```bash
.\mvnw spring-boot:run "-Dspring-boot.run.profiles=dev"
```

```bash
.\mvnw spring-boot:run "-Dspring-boot.run.profiles=prod"
```

### Примечания по профилям

- Профиль **dev** автоматически наполняет хранилище тестовыми данными при запуске
- При использовании профиля **prod** данные хранятся в файловой БД H2
- Переключение между профилями требует перезапуска приложения
- По умолчанию, если профиль не указан, используется **dev**

## Task 4: Docker Support

### Запуск приложения в Docker

Для запуска приложения в Docker-контейнере выполните:

```bash
docker-compose up -d
```

### Просмотр логов контейнера

```bash
docker-compose logs -f app
```

### Остановка контейнеров

```bash
docker-compose down
```

### Пересборка и перезапуск

```bash
docker-compose up -d --build
```

### Особенности Docker-конфигурации

- Для сборки используется многоэтапный Dockerfile
- Приложение запускается с профилем `prod`
- Данные хранятся в файловой H2 базе данных
- База данных сохраняется в Docker-volume `h2-data`
- Порт 8080 проброшен на хост-машину
- Логи сохраняются в директорию `./logs`

### Хранение данных

При использовании Docker-конфигурации:
- Данные сохраняются между перезапусками контейнеров
- Для хранения используется Docker Named Volume (`h2-data`)
- Все изменения в данных сохраняются на диске

Для удаления всех данных используйте:
```bash
docker-compose down -v
```

## Task 6: Кэширование с Redis

### Основные изменения

1. **Добавление Redis в проект**
   - Добавлены зависимости Spring Boot Starter Data Redis и Spring Boot Starter Cache
   - Настроен Redis как провайдер кэширования
   - Добавлен сервис Redis в docker-compose.yml

2. **Конфигурация кэширования**
   - Создан CacheConfig для настройки Redis и Spring Cache
   - Настроены разные TTL (время жизни) для разных типов кэшей:
     - `userTasks` - 5 минут (все задачи пользователя)
     - `pendingTasks` - 2 минуты (невыполненные задачи)
     - `taskById` - 15 минут (отдельные задачи)
   - Настроена сериализация данных в JSON формат

3. **Автоматическое кэширование**
   - Добавлены аннотации `@Cacheable` для методов получения задач
   - Добавлены аннотации `@CacheEvict` для автоматической очистки кэша при изменениях
   - Кэш автоматически очищается при создании и удалении задач

### Принцип работы кэширования

1. **Поиск в кэше** - при запросе данных сначала проверяется Redis
2. **Загрузка из БД** - если данных нет в кэше, они загружаются из MariaDB
3. **Сохранение в кэш** - полученные данные автоматически сохраняются в Redis
4. **Автоматическое истечение** - данные удаляются из кэша по истечении TTL
5. **Очистка при изменениях** - кэш очищается при создании/удалении задач

### Запуск с Redis

Для локального запуска с Redis:
```bash
docker-compose up redis mariadb -d
.\mvnw spring-boot:run
```

Для запуска в Docker:
```bash
docker-compose up -d
```

### Мониторинг кэша

Для просмотра содержимого Redis:
```bash
docker exec -it <redis_container_name> redis-cli
```

Полезные команды Redis:
```bash
KEYS *              # Показать все ключи
TTL <key>           # Показать время жизни ключа
GET <key>           # Получить значение по ключу
FLUSHALL            # Очистить весь кэш
```

## Task 7: Система сообщений (Kafka)

### Основные изменения

1. **Добавление Apache Kafka**
   - Добавлены зависимости Spring Kafka в pom.xml
   - Настроен Kafka и Zookeeper в docker-compose.yml
   - Создана конфигурация KafkaConfig для producer и consumer

2. **Модель событий**
   - Создан класс `TaskCreatedEvent` для передачи данных о созданных задачах
   - Настроена JSON сериализация/десериализация событий

3. **Асинхронная обработка сообщений**
   - Создан `KafkaMessageProducer` для публикации событий в Kafka
   - Создан `KafkaMessageListener` для асинхронной обработки событий
   - TaskService теперь публикует события вместо прямого создания уведомлений

4. **Разделение ответственности**
   - NotificationService получает обновления ТОЛЬКО через message broker
   - Убрана прямая зависимость TaskService от NotificationService для создания задач
   - Уведомления создаются асинхронно через Kafka listener

### Архитектура messaging

```
TaskService -> KafkaMessageProducer -> Kafka Topic -> KafkaMessageListener -> NotificationService
```

1. **Создание задачи** - TaskService сохраняет задачу в БД
2. **Публикация события** - Отправляется TaskCreatedEvent в Kafka топик
3. **Асинхронная обработка** - KafkaMessageListener получает событие
4. **Создание уведомления** - Автоматически создается уведомление пользователю

### Конфигурация Kafka

- **Топик**: `task-created` для событий создания задач
- **Consumer Group**: `task-manager-group` для обработки сообщений
- **Сериализация**: JSON для передачи объектов TaskCreatedEvent
- **Надежность**: Настроена гарантированная доставка сообщений

## Task 8: Планировщик и асинхронные задачи

### Основные изменения

1. **Включение планировщика и асинхронности**
   - Добавлены аннотации `@EnableScheduling` и `@EnableAsync` в главный класс приложения
   - Создана конфигурация AsyncConfig для настройки ThreadPoolTaskExecutor

2. **Периодическая проверка просроченных задач**
   - Создан ScheduledTaskService с методом `@Scheduled` для проверки каждые 5 минут
   - Добавлен метод `findOverdueTasks()` в TaskRepository для поиска просроченных задач
   - Реализована логика в JpaTaskRepository и InMemoryTaskRepository

3. **Асинхронная обработка событий**
   - Создана модель `TaskOverdueEvent` для событий просроченных задач
   - Добавлен метод `@Async` для фоновой обработки найденных просроченных задач
   - Расширен KafkaMessageProducer для публикации событий о просроченных задачах

4. **Обновление системы сообщений**
   - Обновлена конфигурация Kafka для работы с разными типами событий
   - Добавлен новый топик `task-overdue` для событий просроченных задач
   - Расширен KafkaMessageListener для обработки событий просроченных задач

5. **Асинхронные уведомления**
   - Добавлен метод `createNotificationsAsync()` в NotificationService
   - Настроена асинхронная обработка с использованием ThreadPoolTaskExecutor

### Принцип работы планировщика

1. **Периодическая проверка** - каждые 5 минут запускается проверка просроченных задач
2. **Поиск просроченных задач** - находятся все задачи с targetDate < текущее время
3. **Асинхронная обработка** - найденные задачи обрабатываются в фоновом потоке
4. **Публикация событий** - для каждой просроченной задачи отправляется TaskOverdueEvent в Kafka
5. **Создание уведомлений** - автоматически создаются уведомления пользователям

### Конфигурация асинхронности

- **Core Pool Size**: 2 потока
- **Max Pool Size**: 5 потоков  
- **Queue Capacity**: 100 задач
- **Thread Name Prefix**: "AsyncTask-"

### Новые топики Kafka

- **task-overdue** - события просроченных задач
- **task-created** - события создания задач (существующий)

### Мониторинг планировщика

В логах приложения можно отслеживать:
- Запуск проверки просроченных задач каждые 5 минут
- Количество найденных просроченных задач
- Асинхронную обработку событий
- Создание уведомлений о просроченных задачах
